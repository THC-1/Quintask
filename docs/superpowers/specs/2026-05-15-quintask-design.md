# Quintask 在线任务分配平台设计文档

日期：2026-05-15

## 1. 项目目标

Quintask 是一个面向 5 人内部项目小组的在线任务分配平台。它不做公开注册、不做多团队 SaaS，也不追求复杂项目管理能力。第一版目标是让小组内部可以清楚地分配任务、跟踪进度、记录协作过程，并让老师通过只读账号看到每位同学的工作任务和项目整体进展。

系统重点解决三个问题：

- 负责人能把任务拆清楚、分配清楚、验收清楚。
- 组员能看到自己的任务、更新进度、提交验收。
- 老师能只读查看项目进度，以及每个成员承担了哪些工作。

## 2. 适用范围

第一版只服务一个内部项目小组。系统保留 `Project` 数据表，但默认只有一个当前项目，不做多组织、多团队、多租户。

第一版包含：

- 混合看板式任务流转。
- 负责人、成员、老师游客三种角色。
- 里程碑管理。
- 主任务管理。
- 轻量子任务 checklist。
- 轻量任务依赖。
- 标签分类。
- 评论和进度记录。
- 团队工作量展示。
- 负责人维护成员账号。

第一版不包含：

- 文件上传和附件管理。
- 公开注册。
- 多团队 SaaS。
- 工时填报。
- 甘特图。
- 自动排期。
- 复杂审批流。
- 子任务嵌套。

## 3. 角色设计

### 负责人 OWNER

负责人拥有全部写权限：

- 创建、编辑、删除任务。
- 分配任务给成员。
- 调整任务状态。
- 创建、编辑、删除里程碑。
- 创建、编辑标签。
- 新增、禁用成员账号。
- 重置成员密码。
- 验收任务，将任务从 `待验收` 改为 `已完成`，或退回 `进行中`。
- 查看全部统计和团队工作量。

### 组员 MEMBER

组员负责执行和反馈进度：

- 查看所有任务、里程碑和团队工作量。
- 创建建议任务。
- 查看自己和团队成员的任务状态。
- 更新分配给自己的主任务状态。
- 维护自己任务里的子任务完成情况。
- 给任意任务添加评论或进度记录。
- 将自己的任务提交到 `待验收`。
- 不能直接将任务改为 `已完成`。
- 不能管理成员、里程碑、标签等全局配置。

### 老师游客 TEACHER

老师游客使用固定账号登录，只读查看：

- 查看 Dashboard。
- 查看任务看板。
- 查看任务详情。
- 查看里程碑。
- 查看团队工作量。
- 不能创建、编辑、删除任何数据。
- 不能发表评论或进度记录。

## 4. 任务层级规则

系统只有一种真正进入项目主线的任务：`Task`。

`Task` 是看板、统计、里程碑、依赖关系和负责人验收的唯一对象。

`Subtask` 只是任务详情里的轻量 checklist：

- 不进入看板。
- 不参与主工作量统计。
- 不挂接里程碑。
- 不允许再拆子任务。
- 不允许设置任务依赖。
- 不允许单独评论。
- 只有 `未完成 / 已完成` 两种状态。

成员可以创建任务，但成员创建的任务属于建议任务：

- 默认状态为 `待处理`。
- 可以没有执行人，或默认建议执行人为创建者。
- 负责人确认后再分配执行人、优先级和里程碑。
- 成员不能把未被负责人确认分配的建议任务推进到 `进行中`。

项目主线保持为：

`里程碑 -> 主任务 -> 子任务 checklist / 评论 / 依赖提示`

## 5. 任务状态流转

任务采用混合看板式流转：

- `TODO`：待处理。
- `IN_PROGRESS`：进行中。
- `IN_REVIEW`：待验收。
- `DONE`：已完成。

流转规则：

- 负责人可以调整所有任务状态。
- 成员只能推进分配给自己的任务。
- 成员可以将自己的任务从 `待处理` 推进到 `进行中`。
- 成员可以将自己的任务从 `进行中` 提交到 `待验收`。
- 只有负责人可以将 `待验收` 改为 `已完成`。
- 负责人可以将 `待验收` 退回 `进行中`。

依赖采用弱限制：

- 如果依赖任务未完成，当前任务可以进入 `进行中`。
- 如果依赖任务未完成，当前任务不能进入 `待验收` 或 `已完成`。

## 6. 核心页面

### 6.1 登录页

用户输入账号和密码登录。登录成功后根据角色进入系统。

### 6.2 Dashboard 首页

Dashboard 是老师和负责人最先看到的项目概览：

- 整体完成率。
- 当前里程碑。
- 阻塞任务数量。
- 待验收任务数量。
- 最近进度记录。
- 成员任务概览入口。

老师登录后默认进入 Dashboard。

### 6.3 Task Board 任务看板

任务看板按状态分为四列：

- 待处理。
- 进行中。
- 待验收。
- 已完成。

任务卡片展示：

- 标题。
- 执行人。
- 优先级。
- 截止时间。
- 所属里程碑。
- 标签。
- 是否存在未完成依赖。

负责人可创建、分配、调整任务。成员主要更新自己的任务。老师只读查看。

### 6.4 Task Detail 任务详情

任务详情展示和编辑任务主体信息：

- 标题。
- 描述。
- 状态。
- 优先级。
- 执行人。
- 创建人。
- 验收人。
- 截止时间。
- 所属里程碑。
- 标签。
- 依赖任务。
- 子任务 checklist。
- 评论和进度记录。

### 6.5 Milestones 里程碑

里程碑表示项目阶段，例如：

- 需求分析。
- 原型设计。
- 前端开发。
- 后端开发。
- 联调测试。
- 答辩准备。

每个里程碑展示：

- 标题。
- 描述。
- 起止时间。
- 状态。
- 关联任务数量。
- 完成任务数量。
- 完成率。

### 6.6 Team Workload 团队工作量

团队工作量页面是老师查看每位同学贡献的核心页面。

每个成员展示：

- 负责的主任务总数。
- 待处理任务数。
- 进行中任务数。
- 待验收任务数。
- 已完成任务数。
- 阻塞任务数。
- 当前负责的任务列表。
- 参与过的里程碑。
- 最近提交的评论或进度记录。
- 优先级分布。

工作量统计只统计主任务。子任务完成情况可以作为任务详情中的辅助信息展示，但不单独计入主工作量。

系统不做工时统计，避免成员为了展示贡献而填写不可靠数据。

### 6.7 Members 成员管理

成员管理只对负责人开放：

- 新增成员账号。
- 禁用成员账号。
- 重置成员密码。
- 查看成员角色。

老师账号固定为只读游客账号。

## 7. 数据模型

### 7.1 User

用于存储负责人、组员和老师游客。

字段：

- `id`
- `name`
- `username`
- `passwordHash`
- `role`
- `isActive`
- `createdAt`

角色枚举：

- `OWNER`
- `MEMBER`
- `TEACHER`

### 7.2 Project

第一版只有一个项目，但保留项目表以保持结构完整。

字段：

- `id`
- `name`
- `description`
- `createdAt`
- `updatedAt`

### 7.3 Milestone

字段：

- `id`
- `projectId`
- `title`
- `description`
- `startDate`
- `dueDate`
- `status`
- `sortOrder`

状态枚举：

- `PLANNED`
- `ACTIVE`
- `DONE`

### 7.4 Task

字段：

- `id`
- `projectId`
- `milestoneId`
- `title`
- `description`
- `status`
- `priority`
- `assigneeId`
- `creatorId`
- `reviewerId`
- `dueDate`
- `completedAt`
- `createdAt`
- `updatedAt`

状态枚举：

- `TODO`
- `IN_PROGRESS`
- `IN_REVIEW`
- `DONE`

优先级枚举：

- `LOW`
- `MEDIUM`
- `HIGH`

### 7.5 TaskDependency

用于表示主任务之间的依赖。

字段：

- `id`
- `taskId`
- `dependsOnTaskId`

规则：

- `taskId` 表示当前任务。
- `dependsOnTaskId` 表示当前任务依赖的前置任务。
- 只能在主任务之间建立依赖。

### 7.6 Subtask

用于任务详情里的轻量 checklist。

字段：

- `id`
- `taskId`
- `title`
- `isDone`
- `assigneeId`
- `sortOrder`

### 7.7 TaskComment

评论和进度记录共用一张表。

字段：

- `id`
- `taskId`
- `authorId`
- `content`
- `type`
- `createdAt`

类型枚举：

- `COMMENT`
- `PROGRESS`
- `REVIEW`

### 7.8 Tag

字段：

- `id`
- `name`
- `color`
- `createdAt`

### 7.9 TaskTag

任务和标签的多对多关联表。

字段：

- `taskId`
- `tagId`

## 8. 技术架构

前端：

- Vue 3
- Vite
- TypeScript
- Vue Router
- Pinia
- fetch 或轻量 API client

后端：

- Hono
- Prisma
- MySQL
- JWT 鉴权

建议后端模块：

- `auth`：登录、当前用户、修改密码。
- `users`：成员管理。
- `projects`：项目信息。
- `milestones`：里程碑 CRUD。
- `tasks`：任务 CRUD、状态流转、依赖校验。
- `comments`：评论和进度记录。
- `dashboard`：首页统计。
- `workload`：成员工作量统计。
- `tags`：标签管理。

建议前端路由：

- `/login`
- `/dashboard`
- `/board`
- `/tasks/:id`
- `/milestones`
- `/workload`
- `/members`

## 9. API 设计

所有接口使用 `/api` 前缀，并返回 JSON。

认证：

- `POST /api/auth/login`
- `GET /api/auth/me`

成员：

- `GET /api/users`
- `POST /api/users`
- `PATCH /api/users/:id`

项目：

- `GET /api/projects/current`

里程碑：

- `GET /api/milestones`
- `POST /api/milestones`
- `PATCH /api/milestones/:id`

任务：

- `GET /api/tasks`
- `POST /api/tasks`
- `GET /api/tasks/:id`
- `PATCH /api/tasks/:id`
- `POST /api/tasks/:id/status`

评论：

- `POST /api/tasks/:id/comments`

子任务：

- `POST /api/tasks/:id/subtasks`
- `PATCH /api/subtasks/:id`

统计：

- `GET /api/dashboard`
- `GET /api/workload`

标签：

- `GET /api/tags`
- `POST /api/tags`

## 10. 错误处理

统一错误格式：

```json
{
  "error": {
    "code": "TASK_DEPENDENCY_BLOCKED",
    "message": "该任务存在未完成的依赖任务，暂不能提交验收。"
  }
}
```

常见错误码：

- `UNAUTHORIZED`：未登录。
- `FORBIDDEN`：权限不足。
- `NOT_FOUND`：数据不存在。
- `VALIDATION_ERROR`：表单校验失败。
- `TASK_DEPENDENCY_BLOCKED`：依赖任务未完成。
- `INVALID_TASK_TRANSITION`：任务状态不允许流转。

## 11. 测试策略

后端优先测试业务规则：

- 登录成功和失败。
- 角色权限拦截。
- 成员不能修改他人任务状态。
- 老师不能执行写操作。
- 依赖未完成时不能提交验收。
- 负责人可以验收任务。
- 工作量统计按成员正确汇总。

前端重点测试关键流程：

- 登录后按角色进入页面。
- 老师只读状态下不显示编辑按钮。
- 任务看板按状态展示。
- 任务详情展示子任务、依赖和评论。
- 成员工作量页面显示每个人的任务列表和数量。

## 12. 第一版完成标准

第一版完成后，应满足以下标准：

- 负责人可以创建成员账号并分配任务。
- 成员可以更新自己的任务进度并提交验收。
- 负责人可以验收或退回任务。
- 老师可以通过固定游客账号只读查看进度。
- 看板能按任务状态展示项目任务。
- 里程碑能展示阶段任务和完成率。
- 团队工作量页面能展示每位成员承担的任务和状态分布。
- 子任务和依赖关系不会脱离主任务体系，保持轻量。
